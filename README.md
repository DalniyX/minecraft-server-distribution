## 🏗 УРОВЕНЬ 1: АРХИТЕКТУРА И ГИБРИДНЫЙ МОСТ `pywebview`

Лаунчер использует гибридную двухслойную архитектуру:
1. **Слой представления (React 18 + Vite + Tailwind CSS + Lucide Icons)**: Рендерится внутри легковесного нативного окна операционной системы с использованием веб-движка. Весь интерфейс, переходы между вкладками, админ-панель, селекторы модов и кастомизация осуществляются на стороне веб-слоя.
2. **Слой вычислений (Python 3.11+ + Pywebview + Requests + Threading)**: Выполняет нативные операции с диском, подсчитывает хэши SHA-256 модов, скачивает файлы в многопоточном режиме с защитой от обрывов, открывает проводник, управляет состоянием нативного окна (сворачивание, максимизация, перетаскивание за кастомный `TitleBar`).

### 🌉 Схема моста Pywebview API
Связь между React и Python бесшовная:
* **Python ➔ React**: Python-код регистрирует класс `Api`. На стороне React все методы класса доступны асинхронно через объект `(window as any).pywebview.api.method_name(args)`.
* **React ➔ Python (Обратные вызовы)**: Для отправки прогресса или завершения задач в реальном времени Python использует метод `window.evaluate_js(f"window.functionName(...)")`.

## 📂 УРОВЕНЬ 2: ПОДРОБНАЯ КАРТА ФАЙЛОВ И МОДУЛЕЙ

Ниже представлена детальная древовидная структура проекта с описанием ответственности каждого конкретного файла.

### 🐍 ЯДРО ЛАУНЧЕРА (PYTHON БЭКЕНД)

* **`/main.py`** — Центральный оркестратор бэкенда.
    * *Ответственность*: Инициализирует окно `pywebview`, регистрирует класс взаимодействия `Api`, обрабатывает перетаскивание окна (`drag_window`), управляет разворачиванием окна, отправляет отзывы в GitHub Issues через зашифрованные PAT-токены, выполняет запросы к Modrinth API для поиска и скачивания модов.
* **`/core/`** — Архитектурные модули для работы с локальной системой.
    * **`config_manager.py`** — Менеджер локальной конфигурации лаунчера.
        * *Ответственность*: Определяет путь к папке игры (по умолчанию `%APPDATA%/.minecraft`). Сохраняет настройки лаунчера, такие как токен авторизации администратора, адрес репозитория модов и токен в системную папку `%LOCALAPPDATA%/DalniyX_Launcher/config.json`.
    * **`file_manager.py`** — Сканер диска и обработчик локальной структуры `.minecraft`.
        * *Ответственность*: Подсчитывает хэши SHA-256 файлов, сравнивает их с манифестом, удаляет неактуальные или запрещенные файлы, управляет подпапками (`mods`, `resourcepacks`, `shaderpacks`, `config`).
    * **`downloaders/`** — Дополнительные низкоуровневые модули загрузок файлов.
* **`/workers/`** — Фоновые потоки выполнения длительных задач.
    * **`download_worker.py`** — Многопоточный загрузчик.
        * *Ответственность*: Осуществляет многопоточное скачивание необходимых файлов, контролирует скорость загрузки, возобновляет скачивание при сбоях сети, сообщает о текущем прогрессе во фронтенд через JS-инъекции.

* **`/src/types.ts`** — Центральный файл типов TypeScript. Содержит все типы данных для манифеста, файлов, настроек Гитхаба и состояний скачивания.

    * *Ответственность*: Эмулирует методы `pywebview` в обычном браузере для быстрой отладки интерфейса без необходимости постоянной компиляции лаунчера.

## 🧬 УРОВЕНЬ 3: СПЕЦИФИКАЦИИ ДАННЫХ И BRIDGE API

### 1. Структура файла сборки (`ModpackFile`)
Используется во фронтенде и бэкенде для репрезентации каждого конкретного мода или ресурс-пака.
```typescript
interface ModpackFile {
  id: string;               // Уникальный идентификатор файла
  name: string;             // Имя файла на диске (например, "sodium-1.2.jar")
  size: number;             // Размер файла в байтах
  sha256: string;           // Хэш сумма для проверки целостности файла
  category: "Core" | "Optimization" | "Content" | "Library" | "ResourcePack" | "ShaderPack" | "Config";
  status: "aktualen" | "update" | "to-delete" | "custom"; // Статус файла относительно сервера
  localSha256?: string;     // Фактический хэш файла на жестком диске
  enabled?: boolean;        // Флаг активности мода
  description?: string;     // Короткое описание назначения файла
  version?: string;         // Версия мода
  slug?: string;            // Идентификатор мода на Modrinth
}
```

### 2. Структура Манифеста (`ModpackManifest`)
Хранится в вашем GitHub репозитории сборки как файл `manifest.json`.
```typescript
interface ModpackManifest {
  launcher_version: string; // Требуемая версия лаунчера для этой сборки
  admin_pwd_hash: string;   // Хэш-сумма пароля администратора (SHA-256)
  minecraft_version: string;// Версия Minecraft (например, "1.20.1")
  modrinth_mc_version?: string; // Техническая версия игры для запросов к Modrinth
  loader_type: "Forge" | "Fabric" | "NeoForged"; // Загрузчик модов
  modrinth_search_enabled?: boolean; // Разрешить ли поиск по Modrinth в лаунчере
  ai_changelog?: string;    // Накопительный чейнжлог, сгенерированный ИИ
  accent_color?: string;    // HEX код акцентного свечения лаунчера
  primary_color?: string;   // HEX код вспомогательного цвета оформления
  background_image?: string;// URL ссылки на фоновое изображение сборки
  bg_blur?: number;         // Сила размытия фона (в пикселях)
  files: Array<{            // Массив файлов сборки
    name: string;
    size: number;
    sha256: string;
    category: string;
    description?: string;
    version?: string;
    slug?: string;
  }>;
}
```

### 3. Конфигурация Гитхаба (`GitHubConfig`)
Используется администратором для соединения лаунчера со своим репозиторием модов.
```typescript
interface GitHubConfig {
  owner: string;            // Никнейм владельца репозитория
  repo: string;             // Название репозитория сборки
  token: string;            // Персональный токен доступа (PAT) с правами repo
  branch: string;           // Рабочая ветка (по умолчанию "main")
  manifestPath: string;     // Путь к файлу манифеста (по умолчанию "manifest.json")
}
```

### 4. Методы Python Bridge API (`pywebview.api`)
Список функций, экспортируемых классом `Api` в файле `main.py`, которые React вызывает напрямую:

1. **`start_sync()`** — Запускает фоновый поток многопоточной синхронизации модов с Гитхабом.
2. **`open_file_location(filename, category)`** — Открывает системный проводник Windows, переходит в соответствующую подпапку (например, `.minecraft/mods` или `.minecraft/shaderpacks`) и автоматически подсвечивает нужный файл.
3. **`check_file_exists(filename, category)`** — Проверяет, присутствует ли указанный файл в локальном каталоге Minecraft.
4. **`fetch_modrinth_mod(slug, mc_version, loader)`** — Обращается к серверам Modrinth, скачивает указанный мод во временную системную папку, вычисляет его размер и хэш-сумму SHA-256 для последующего добавления администратором в сборку.
5. **`check_modrinth_updates(slugs, mc_version, loader)`** — Массово проверяет наличие свежих версий для списка модов по их Modrinth-слагам.
6. **`save_github_config(user, repo, token, gemini_key)`** — Сохраняет параметры интеграции с GitHub в локальный конфиг.
7. **`load_github_config()`** — Читает сохраненные данные репозитория и ключи ИИ.
8. **`get_build_settings()`** — Возвращает запеченные параметры клиента (версия сборки, название клиента) из файла `build_settings.json`.
9. **`select_mods()`** — Открывает стандартный проводник ОС для выбора файлов `.jar` или `.zip` для импорта их администратором.
10. **`upload_to_github(user, repo, token, manifest_json_str)`** — Опубликовывает обновленный файл `manifest.json` в репозиторий.
11. **`upload_mod_to_github(user, repo, token, local_file_path, github_filename, category, base64_data)`** — Бесшовно загружает файл мода в соответствующую подпапку репозитория на GitHub через REST API.
12. **`drag_window()`, `minimizeApp()`, `toggleMaximizeApp()`, `closeApp()`** — Нативные оконные методы управления окном.

### 5. Локальный кэш лаунчера (`config.json`)
Файл конфигурации лаунчера располагается по пути:
`%LOCALAPPDATA%/DalniyX_Launcher/config.json`
Он содержит локальные системные пути и ключи, которые не должны утекать в публичные репозитории:
```json
{
    "minecraft_path": "C:\\Users\\Username\\AppData\\Roaming\\.minecraft",
    "github_user": "your_github_username",
    "github_repo": "your_minecraft_repo",
    "github_token": "github_pat_...",
    "gemini_api_key": "AIzaSy..."
}
```

```css
  --theme-accent: #5e00f5; /* Насыщенный фиолетовый */
  --theme-primary: #007a6e; /* Глубокий бирюзовый */
}
```

### 2. Специфические правила отрисовки UI элементов

#### 2.1 Настройки Темной Темы (Default)
* **Задники окон**: Отрисовываются глубоким космическим цветом `#08080c` с наложением полупрозрачной подложки `rgba(6, 6, 9, 0.35)` поверх кастомных обоев, что дает потрясающее благородное свечение без пересвета.
* **Рамки и Бордеры**: Имеют мягкий белый цвет с высокой степенью прозрачности `rgba(255,255,255,0.08)`. Это подчеркивает грани, но не засоряет экран яркими линиями.

#### 2.2 Настройки Светлой Темы (Light Mode)
* **Задники окон**: Переходят на высокоплотную, непрозрачную матовую подложку `rgba(255, 255, 255, 0.82)` с экстремальной силой размытия заднего фона `blur(28px)`. Это изолирует контент от пестроты обоев сборки и гарантирует профессиональный журнальный вид.
* **Рамки и Бордеры**: Получают белую прорисовку повышенной контрастности `rgba(255, 255, 255, 0.65)` для создания эффекта премиальной стеклянной огранки.

#### 2.3 Эффекты стекла и затемнений
В файле `src/index.css` определены автоматические трансляторы классов:
* Любые модальные подложки в светлой теме (`.light .bg-black/70` и т.д.) автоматически переводятся в изящные светлые оверлеи с размытием `blur(10px)`.
* Кнопки при наведении на светлом фоне плавно меняют свой бэкграунд на плотный белый цвет без избыточного мерцания.

#### 2.4 Правила читаемости шрифтов и текста
* В светлой теме жестко запеченные белые классы текста Tailwind автоматически перенаправляются на глубокий контрастный цвет заголовков `--text-title` (`#0f172a`), а серые классы `text-zinc-200/300` заменяются на `--text-main` (`#1e293b`).
* Цвета баджей, такие как оранжевый, красный, зеленый и синий, в светлой теме автоматически становятся на 40% темнее и насыщеннее (например, ярко-красный `text-red-400` становится темно-вишневым `#be123c`), гарантируя 100% соответствие стандартам доступности интерфейсов WCAG AA.

### 🚀 ПЕРВЫЙ ЗАПУСК ИГРОКА
1. Запустите лаунчер. Программа автоматически создаст необходимые скрытые директории конфигурации.
2. Откроется `SplashScreen`, который проверит вашу систему, подключение к интернету и определит путь к нативной сборке Minecraft на вашем жестком диске.
3. В окне игрока нажмите кнопку **«СИНХРОНИЗИРОВАТЬ И ИГРАТЬ»**.
4. Лаунчер сопоставит ваши локальные файлы с облачным манифестом сервера, автоматически удалит устаревшие моды, загрузит недостающие файлы в многопоточном режиме и подготовит игру к запуску.
5. Для тонкой настройки (смена папки игры, переключение языка или включение светлой темы) нажмите на иконку шестеренки **«Настройки»** в левом нижнем углу.

### 🛠 ПЕРВЫЙ ЗАПУСК И НАСТРОЙКА АДМИНИСТРАТОРА

#### Подключение к базе данных / хранилищу GitHub:
1. На главном экране лаунчера нажмите кнопку **«Войти в админ-панель»** в правом верхнем углу.
2. Введите ваш секретный пароль администратора (по умолчанию `admin` или `modpack2026`).
3. Перейдите на вкладку **«Конфигурация»**.
4. Заполните данные вашего GitHub репозитория:
    * **GitHub Owner** — Ваш логин на GitHub.
    * **GitHub Repository** — Название пустого или существующего репозитория (например, `my-cool-modpack`).
    * **GitHub PAT Token** — Персональный токен доступа Гитхаба (генерируется в настройках GitHub -> Developer Settings -> Personal Access Tokens -> Tokens (classic) с галочкой `repo`).
5. Нажмите кнопку **«Сохранить настройки»**. Соединение проверится автоматически, и статус-индикатор вверху загорится зеленым: **`SYNC DB [ACTIVE]`**.

#### Управление файлами сборки:
1. Перейдите на вкладку **«Манифест и Моды»**.
2. **Добавление модов**: Просто перетащите один или несколько файлов `.jar`/`.zip` в зону загрузки "Импорт ресурсов" или нажмите кнопку "Выбрать файлы" для выбора модов через стандартный проводник системы.
3. **Поиск по Modrinth**: Введите название мода в поисковую строку. Лаунчер автоматически отфильтрует результаты именно под версию вашей сборки (например, Fabric 1.20.1). Нажмите кнопку «Найти и Скачать» — лаунчер сам загрузит мод с серверов Modrinth, посчитает хэши и добавит его в список.
4. **Редактирование категорий**: Вы можете изменить категорию для любого файла (например, перевести мод в категорию *«Optimization»*, *«Content»* или *«ResourcePack»*).
5. Нажмите большую кнопку **«Опубликовать Моды»**. Лаунчер самостоятельно зальет все добавленные файлы в ваш репозиторий GitHub и обновит файл `manifest.json`. С этого момента у всех ваших игроков при следующем запуске лаунчер автоматически скачает новые моды!

## 📦 СИНХРОНИЗАЦИЯ И ОБНОВЛЕНИЯ ПО ВОЗДУХУ (OTA)

Лаунчер поддерживает бесшовное фоновое обновление самого себя без необходимости ручного перескачивания пользователем.

### Как выпустить обновление клиента лаунчера для игроков:
1. Перед компиляцией откройте локальный файл `build_settings.json` и увеличьте значение версии (например, `"client_version": "1.2.0"`).
2. Скомпилируйте проект в исполняемый `.exe` файл.
3. Зайдите на страницу вашего глобального репозитория GitHub, перейдите в раздел **Releases** и опубликуйте новый релиз с тегом `v1.2.0`.
4. Прикрепите скомпилированный `.exe` файл к этому релизу и скопируйте *прямую ссылку* на скачивание этого `.exe` файла.
5. Обновите файл `version.json` в корне вашего репозитория обновлений:
   ```json
   {
     "version": "1.2.0",
     "download_url": "https://github.com/.../releases/download/v1.2.0/launcher.exe"
   }
   ```
6. При следующем старте лаунчеры игроков обнаружат новую версию, отобразят красивый чейнжлог релиза и автоматически скачают и заменят исполняемый файл на новый!
